## Context

当前设置页由多个独立 tab 组件构成，每个 tab 自己负责：

- 读取本地配置
- 读取远端配置（部分 tab）
- 构建默认值
- 保存按钮与取消按钮
- 关闭对话框

这种结构在配置数量增加后带来了两个问题：

1. 信息架构不稳定。新能力只能继续新增 tab 或往现有 tab 里线性堆叠。
2. 保存主干分散。不同 tab 的保存语义并不完全相同，但当前 UI 结构并没有明确表达这种差异。

## Goals

- 用统一工作区承载设置项，减少一级导航数量
- 在不改变功能语义的前提下集中管理表单型配置的加载与保存
- 为小屏幕提供更稳健的设置体验
- 为后续继续扩展配置提供稳定的布局骨架

## Non-Goals

- 不改变已有配置字段的业务语义
- 不改变 viewer 契约或分享逻辑
- 不把黑名单等管理型操作硬并入统一“点击保存后才生效”的流程

## Current Save Semantics

### 1. 本地专属配置

以下字段属于本地专属或主要依赖本地配置：

- `serviceApiConfig.token`
- `serviceApiConfig.apiUrl`
- `isNewUIEnabled`
- `isCustomCssEnabled`
- `siyuanConfig.preferenceConfig.fixTitle`

这些字段保存时必须写入完整 `ShareProConfig` 到本地。

### 2. 可同步的 appConfig 配置

以下分散在多个 tab 中的表单项，本质上都在编辑 `appConfig`：

- 个性设置
- 文档设置
- SEO 设置
- 增量分享开关

这些字段当前采用“先保存本地完整配置，再同步 `appConfig` 到服务端”的模式。

### 3. 即时操作型配置

黑名单管理不是普通表单，而是通过 `LocalBlacklistService` 逐项即时执行：

- 添加
- 删除
- 清空
- 搜索与分页

它已经自带自己的持久化和远端同步链路，不应被改造成统一草稿流。

## Decisions

### Decision 1: 使用统一设置工作区，而不是继续扩 tab

设置页将重构为统一工作区，一级导航收敛为稳定的高层分区，例如：

- 常规
- 展示
- 分享
- 增量

这样可以让后续新增配置优先进入分区内 section，而不是继续拉长一级导航。

### Decision 2: 表单型配置改为统一草稿模型

`ShareSetting.svelte` 将成为单一的数据拥有者，负责：

- 一次性加载本地完整配置
- 一次性加载远端 `appConfig`
- 生成统一草稿模型
- 跟踪脏状态
- 提供统一保存入口

各个设置面板将只负责展示与编辑草稿，不再各自直接调用保存或关闭对话框。

### Decision 3: 统一保存出口保持原有持久化顺序

统一保存必须保持现有主干语义：

1. 生成最终 `ShareProConfig`
2. 保存完整配置到本地
3. 如果涉及 `appConfig`，则调用 `syncAppConfig` 同步到服务端
4. 根据结果提示成功或“本地已保存但同步失败”

这意味着本次重构是“保存入口集中化”，不是“保存行为改写”。

### Decision 4: 黑名单管理保持即时操作

黑名单管理会被纳入新的工作区结构，但仍然保持即时操作模型：

- 不依赖统一草稿
- 不等待统一保存按钮
- 不参与整体脏状态

这样可以避免对现有管理流造成回归。

### Decision 5: 布局采用 section/row 与按需展开

为了降低高度与阅读成本：

- 普通设置项使用统一 row 布局
- 相关项放入 section 卡片
- 子配置仅在主开关开启后显示
- 文本类配置单独成组
- 底部操作区采用统一 sticky footer

## Proposed Structure

### 常规

- 注册信息
- 注册码
- 新 UI

### 展示

- 主题
- 自定义 CSS
- 域名与路径
- SEO 信息

### 分享

- 子文档分享
- 引用文档分享
- AI 助手
- 信息栏
- 文档树
- 文档大纲
- 密码保护

### 增量

- 增量分享开关
- 黑名单管理

## Responsive Strategy

- 宽屏保留侧边导航
- 中小屏切换为顶部横向导航或选择器
- 内容区保持独立滚动
- 底部保存操作始终可见

## Risks / Trade-offs

- 风险：集中保存后，某些 tab 原有的独立关闭行为可能被误改
  - Mitigation: 明确区分统一草稿流与即时操作流，并保持黑名单独立
- 风险：远端 `appConfig` 合并时覆盖本地草稿不当
  - Mitigation: 在容器层统一定义加载优先级，并用测试覆盖远端覆盖与默认值回退
- 风险：大改 UI 容易出现布局回归
  - Mitigation: 先提取通用 shell 与 row，再迁移内容，最后清理旧结构

## Test Strategy

- 单测：草稿初始化时本地配置、远端 `appConfig`、默认值的合并
- 单测：统一保存时本地保存与远端同步的调用顺序
- 单测：同步失败时保留“本地已保存”的语义
- 单测：黑名单操作不进入统一草稿保存流
- 组件测试：小屏幕下导航切换不影响当前编辑状态
